# Devplan — cerase-acp v0.1 (PoC milestone M4)

**Status**: working document
**Started**: 2026-05-24
**Umbrella milestone**: `cerase/devplan/poc.md` §M4 — `cerase-bridge` (in-house DM connector) + OpenCode agent + DM round-trip

`cerase-acp` is the in-house Discord-to-ACP bridge for Cerase. It is the
single TypeScript artefact in the Cerase stack and is intentionally split
into its own repository so its build pipeline (npm, tsc, vitest) does not
leak into the otherwise-bash/Python/PHP `cerase/` repo. Cerase consumes
this repo as a published Docker image (`cerase-acp:<tag>`), the same way
it consumes OpenCode and LiteLLM.

v0.1 delivers the PoC slice of M4: DM-only Discord ingestion, ACP v1
stdio session per `(user, agent)`, `[turn_meta]` injection,
crash-recovery, allowlist enforcement, and a `BRIDGE_E2E_TEST` injection
endpoint for automated end-to-end tests in the cerase repo. v0.2+ will
add (post-M4): permission gating UX refinements, multi-platform support
(Slack, Telegram) if Tier-0 customers require it, and richer attachment
handling.

**UX scope** (from `cerase/devplan/poc.md` §M2):
- DM-only Discord. No guild-channel listening, no slash commands, no
  buttons, no "control panel" surface. The user talks like to a
  colleague.
- One long-lived ACP session per `(user, agent)` pair. Survives bridge
  restart transparently.
- Per-turn `[turn_meta: gap=…, lang=…]` block prepended to each
  `session/prompt`, read by the agent's system prompt
  (`cerase/agent-runtime/agent/srv/AGENTS.md`).
- Per-agent `user_id` allowlist enforced by the bridge. Unauthorised DM
  → polite refusal reply, an `event_log` row in Cerase Postgres (M5
  wires the row; v0.1 just emits a structured log line that M5 will
  ingest).

**Inspiration**: structural pattern adapted from
[Open-ACP/OpenACP](https://github.com/Open-ACP/OpenACP) — specifically
the separation of `session-manager` / `prompt-queue` / `turn-context` /
adapter-primitive concepts in `src/core/sessions/` and
`src/core/adapter-primitives/`. We **do not** import OpenACP code or
follow its plugin/monorepo architecture (overkill for a single-platform
PoC). We **do not** adopt its forum-thread / slash-command / button-
permission UX (incompatible with Cerase's colleague-style positioning).
Patterns worth porting: 2000-char Discord message chunking with
continuation markers, per-channel rate-limited send queue, streaming
draft manager that buffers `agent_message_chunk` events and flushes on
sentence-boundary / timer / size threshold, per-session prompt queue
that serialises concurrent user turns through a single ACP child.

## M3 — ACP session manager + prompt queue

**Why**: the long-lived per-`(user, agent)` session is the central
abstraction of the bridge. Everything else (turn-meta, streaming,
Discord adapter) plugs into it. Must be testable in isolation — no
Discord, no LLM, no docker — by talking to a fake ACP child that
replies deterministically to stdio NDJSON.

**Scope**:
- `src/session-manager.ts`: `SessionManager` class with a
  `Map<sessionKey, Session>` registry where `sessionKey =
  ${agent_id}:${user_id}`. Lazy-spawn `opencode acp` via configured
  `spawn.command + spawn.args` (in PoC: `docker exec -i
  cerase-agent-<id> opencode acp`). ACP client wraps stdin/stdout via
  `@agentclientprotocol/sdk@0.22.1`. On child exit: log + remove from
  map; next user turn re-spawns transparently. Idle-timeout via
  `setTimeout` reset on each turn, kills the child after configured
  minutes of inactivity. `prompt(text)` enqueues + returns the streamed
  reply.
- `src/prompt-queue.ts`: per-session FIFO. If user sends turn #2 while
  turn #1 is in flight, queue it. ACP is single-threaded per session.
- `src/__tests__/fake-acp-child.ts`: a tiny Node script that reads
  stdin NDJSON, emits `initialize` response with `protocolVersion: 1`,
  responds to `session/new`/`session/prompt` with deterministic
  fixtures, exits cleanly on `session/cancel`.

**Tasks**:
- [x] `src/session-manager.ts` with the lifecycle above (~180 LOC).
- [x] `src/prompt-queue.ts` (~50 LOC).
- [x] `src/__tests__/fake-acp-child.mjs` test fixture — plain .mjs (no
      TypeScript) so the spawn loop has zero transpilation overhead;
      env knobs FAKE_REPLY / FAKE_CHUNKS / FAKE_CRASH_AFTER_PROMPT /
      FAKE_DELAY_MS_PER_CHUNK cover the test surfaces below.
- [x] `src/session-manager.test.ts`: 8 cases — lazy spawn, reuse on
      second prompt, isolation across (agent, user) keys, transparent
      respawn after crash, FIFO serialisation of concurrent prompts,
      unknown-agent rejection, shutdown() cleanup, vitest fake-timer
      driven idle-timeout kill.
- [x] `src/prompt-queue.test.ts`: 4 cases — FIFO under interleaved
      enqueues, no-overlap invariant, failing handler does not block
      the queue, size() drains to 0.
- [x] PoC permission policy: ACP `requestPermission` calls auto-cancel
      with a structured warning log (M2 UX forbids in-DM buttons; M5

- [x] `CERASE_ACP_LOG_LEVEL` env to control pino verbosity (tests set

**Exit gate**: `npm test` green (27 cases total in v0.1 so far); the

## M4 — Turn-meta + stream buffer + send queue

**Why**: the agent receives a `[turn_meta: gap=…, lang=…]` prefix on
every user turn (M2 system-prompt contract). The Discord side must
chunk 2000-char-limited DM replies. The streaming side must batch
`agent_message_chunk` events so we don't emit 100 Discord messages per
second.

**Scope**:
- `src/turn-meta.ts`: track last-user-message timestamp per session;
  on next turn compute `gap` (minutes/hours/days human-readable); a
  simple language heuristic from the incoming text (cheap regex check
  for common tokens — Italian/English/Spanish/French covers PoC); emit
  `[turn_meta: gap=…, lang=…]\n\n` prefix.
- `src/stream-buffer.ts`: accumulates `agent_message_chunk` events.
  Flushes when (a) a sentence boundary is reached AND ≥200 chars
  accumulated, OR (b) the buffer exceeds 1800 chars (leaving margin
  for chunking), OR (c) a 500ms timer fires after the last chunk and
  the buffer is non-empty. On flush emits the chunk to the send queue.
- `src/send-queue.ts`: per-channel FIFO. Splits messages > 1990 chars
  on sentence/newline boundaries with " ⏎" continuation marker (1990
  to leave 10 chars headroom for the marker). Spaces sends by ≥100ms
  to respect Discord's per-DM rate limit (≈5/sec safe ceiling).

**Tasks**:
- [x] `src/turn-meta.ts` + tests (15 cases). Exports formatGap,
      detectLanguage, makeTurnMetaBlock, TurnMetaTracker. Per-(agent,
      user) state. Language heuristic supports it/en/es/fr; everything
      else → "unknown" (agent's system-prompt rules pick the language
      from the message body anyway).
- [x] `src/stream-buffer.ts` + tests (7 cases). Three flush triggers
      (sentence boundary + size threshold, hard cap, idle timer). End-
      of-turn flush via end(). All recognised terminators tested
      (./!/?/newline).
- [x] `src/send-queue.ts` + tests (8 cases). chunkForDiscord() exported
      separately for reuse: prefers newline > sentence > hard-split
      boundaries; non-final chunks carry a " ⏎" marker. SendQueue
      ensures FIFO + ≥minIntervalMs spacing + auto-chunking + survives
      send() throws. drain() resolves only when no chunks pending and
      no in-flight send.

**Exit gate**: `npm test` green (56 cases total in v0.1 so far); the
chunker round-trips text >5000 chars losslessly across multiple
chunks. ✅ CLOSED 2026-05-24.

**Effort**: 1 day.

## M5 — Discord adapter + index.ts + test-injection endpoint

**Why**: wires the pieces from M2..M4 together behind a `discord.js`
client per agent and adds the `BRIDGE_E2E_TEST` HTTP endpoint that
cerase's `tests/e2e-discord/` (B5) uses to drive end-to-end tests
without needing real Discord traffic.

**Scope**:
- `src/discord-adapter.ts`: one `discord.js` `Client` per configured
  agent. `IntentsBitField` set to `Guilds | DirectMessages |
  MessageContent | DirectMessageReactions` (no guild-channel intents).
  `messageCreate` handler: drop bot's own messages, drop non-DM
  messages, check allowlist → either route to session manager
  (allowed) or send polite refusal (not allowed). Reply via the user's
  `DMChannel`. Attachments downloaded to
  `${workdir}/uploads/<turn-id>/` and noted in the user-turn text.
- `src/test-injection.ts`: tiny Fastify (or Node `http.createServer`)
  HTTP server listening on `:7474` ONLY when `BRIDGE_E2E_TEST=1`.
  `POST /_test/inject` accepts `{ agent_id, user_id, text }` and feeds
  the message through the same dispatcher path the Discord adapter
  uses, bypassing only the Discord client. `GET /_test/last-reply` for
  the most recent reply per `(agent_id, user_id)`. Never enabled in
  the published image's default mode.
- `src/index.ts`: load config → spawn one `DiscordAdapter` per agent →
  start `SessionManager` → wire `StreamBuffer` + `SendQueue` →
  conditionally start `TestInjectionServer`. Structured logging via
  pino. SIGTERM handler: stop accepting new messages, drain in-flight
  prompts, kill children, exit 0.

**Tasks**:
- [x] `src/dispatcher.ts`: pure Discord-agnostic pipeline (allowlist →
      session-manager → stream-buffer → send-queue). The unit of
      behavioural truth — Discord adapter and test-injection both
      drive the same dispatcher. Resolves "send target" through an
      injected callback so the dispatcher itself knows nothing about
      Discord channels.
- [x] `src/discord-adapter.ts`: thin discord.js glue. One Client per
      agent (DirectMessages + MessageContent + Guilds intents,
      Channel + Message partials), drops bot/guild messages, caches
      per-user DM channels for fast multi-chunk reply delivery. Lean
      enough to verify behaviourally via cerase's e2e-discord tier
      and BRIDGE_E2E_TEST rather than via discord.js mocks.
- [x] `src/test-injection.ts`: plain `node:http` server bound to
      127.0.0.1 (or 0.0.0.0 if needed by compose). POST /_test/inject
      drives Dispatcher.handleMessage and AWAITS completion (tests
      need determinism — production Discord path also awaits). GET
      /_test/last-reply returns the accumulated chunks for a (agent,
      user) pair with " ⏎" markers stripped.
- [x] `src/index.ts`: wires config → SessionManager + TurnMetaTracker
      → adapter table → Dispatcher (resolves send-target via the
      adapter table). Conditionally starts the test-injection server
      when BRIDGE_E2E_TEST=1. SIGINT/SIGTERM → graceful shutdown
      (stop discord clients, kill ACP children, close test server).
- [x] `src/dispatcher.test.ts` (5 cases) + `src/test-injection.test.ts`
      (4 cases) — driven against the fake-ACP child fixture.

**Exit gate**: `npm run build && node dist/index.js` exits cleanly
(rc=1) on missing config with an operator-readable fatal log; full
v0.1 suite remains green at 65/65; the BRIDGE_E2E_TEST endpoint

## M7 — Standalone CLI + bash wrapper

**Why**: the bridge daemon talks Discord on one side and ACP on the
other; both surfaces are slow to spin up for ad-hoc debugging. A
standalone CLI lets developers exercise the same pipeline
(allowlist → turn-meta → session-manager) for a single round-trip
against a configured agent (the fake-acp-child fixture for local
testing, or a real `opencode acp` once wired) without launching
Discord, compose, or even the test-injection HTTP server. A thin bash
wrapper (`scripts/cerase-acp-cli`) keeps the TypeScript surface lean
by handling the REPL loop and the remote-daemon `inject` case outside
the build artefact — TS stays a single-shot tool, bash adds the
ergonomics.

**Scope**:
- `src/cli.ts`: one subcommand for v0.1 — `prompt`. Flags:
  `--config <path>` (default `agents.yaml` in cwd),
  `--agent <id>` (required), `--user <discord-user-id>` (required),
  positional prompt text. Streams the reply to stdout as ACP
  `agent_message_chunk` updates arrive (no Discord-style chunking
  / rate-limiting — the Dispatcher's StreamBuffer + SendQueue are
  bypassed because the CLI's "channel" is just stdout). Exit 0 on
  success (including the polite-refusal path for unauthorised users —
  refusal is a valid response, not an error). Exit 1 on config errors,
  unknown agent, or ACP failures.
- `src/cli.test.ts`: vitest cases covering the prompt subcommand
  against the fake-acp-child fixture (happy path streaming, refusal
  for unauthorised user, unknown agent → exit 1 + clear message,
  missing required flag → exit 1, help text).
- `scripts/cerase-acp-cli`: bash wrapper (~80 LOC) with three
  subcommands:
    - `prompt …`  — passes args to `node dist/cli.js prompt`
    - `repl …`    — `while read` loop, each line invokes the TS CLI
      (per-turn child spawn; for testing bridge plumbing this is fine,
      and mem0+workspace handles agent-level continuity when running
      against real OpenCode)
    - `inject --remote URL --agent X --user Y "text"` — `curl POST
      /_test/inject` then `curl GET /_test/last-reply`, printing the
      captured reply text. Useful for diagnosing a running bridge
      daemon without standing up a TS process.
- Shared helper: extract `pickRefusalMessage(text)` from
  `dispatcher.ts` so both Dispatcher and CLI use one source of truth
  for the polite-refusal copy.

**Tasks**:
- [x] `src/cli.ts` (~135 LOC) — argv parsing hand-rolled (no
      commander/yargs dep), config load, allowlist gate, turn-meta
      prefix, SessionManager round-trip, stdout streaming via the
      `onUpdate` callback (no Dispatcher StreamBuffer / SendQueue —
      stdout doesn't need Discord-shaped chunking). Graceful shutdown
      via `mgr.shutdown()` in `finally`.
- [x] Extracted `pickRefusalMessage(text)` from `dispatcher.ts`; both
      Dispatcher and CLI now call it — single source of truth for the
      polite-refusal copy.
- [x] `src/cli.test.ts` (8 cases) — streaming happy path, refusal
      path, unknown agent, missing required flag, missing prompt text,
      missing config file, --help, unknown subcommand.
- [x] `scripts/cerase-acp-cli` (~115 LOC bash) — `prompt` / `repl` /
      `inject`. ANSI-C $'...' quoting on color constants so heredoc
      usage interpolates correctly.
- [x] `src/cli-wrapper.test.ts` — vitest spawning the bash script:
      `--help`, unknown subcommand, `prompt` dispatches to TS,
      `repl` two-turn stdin loop, `inject` against an in-memory
      `http.createServer` (echo daemon that upper-cases the prompt),
      `inject` against a blackhole 404 daemon.
- [x] README "Debug CLI" section with example invocations + the
      `CERASE_ACP_LOG_LEVEL=silent` recipe for silencing pino logs
      that interleave with the streamed reply.

**Exit gate**: `npm test` green (87/87); manual end-to-end via the

## M8 — Bridge resilience under failing Discord logins (test-mode)

**Why**: discovered while debugging M7's CLI demo — when `index.ts`
runs `Promise.all([...adapters].map(a => a.start()))` and ALL discord
clients fail (e.g. fake tokens in dev), the rejection propagates
through `main()`, the process exits with rc=1, and the optional
`BRIDGE_E2E_TEST` HTTP server never gets a chance to bind. This breaks
the developer's "spin up the daemon with fake tokens just to talk to
the test endpoint" use case. In production we still want to fail-fast,
because a failed login there is a real config error.

**Scope**:
- Extract the wiring from `src/index.ts` into a new `src/bridge.ts`
  exporting `runBridge(opts) → Promise<{ shutdown }>`. `opts` accepts
  an `adapterFactory` for dependency injection so tests don't need to
  hit real Discord.
- In test mode (`opts.bridgeE2eTest === true`):
  - Start the test-injection HTTP server FIRST, before any adapter.
  - Wrap each adapter `start()` in its own try/catch — log failures,
    do not reject the outer promise.
- In production mode: keep current behaviour — `Promise.all` fans out
  the adapter starts; any rejection bubbles up.
- `src/index.ts` becomes the thin entry point: read env, call
  `runBridge`, wire SIGINT/SIGTERM to the returned shutdown handle.

**Tasks**:
- [x] `src/bridge.ts` — `runBridge(opts)` with `createAdapter`
      injection for tests. Test-mode and production-mode wiring split
      cleanly.
- [x] Secondary bug discovered + fixed via regression test: one shared
      dispatcher pointed its send-target at the Discord adapter,
      meaning `/_test/inject` replies tried to flow back through a
      not-logged-in Discord client (the send-queue's error handler
      swallowed the failure silently → GET `/_test/last-reply` returned
      404). Fix: two dispatchers sharing SessionManager +
      TurnMetaTracker, each with its own send-target. The discord
      dispatcher routes to DM channels; the test-injection dispatcher
      routes to `server.recordReply()`. Forward-reference for the test
      server.
- [x] `src/bridge.test.ts` — 6 cases:
      (a) BRIDGE_E2E_TEST=true + all adapters reject start() → resolves
          + test-injection server reachable;
      (b) BRIDGE_E2E_TEST=true + one adapter succeeds + one rejects →
          still resolves;
      (c) BRIDGE_E2E_TEST=false + an adapter rejects → rejects with
          best-effort teardown of partial state;
      (d) BRIDGE_E2E_TEST=false + all succeed → resolves, no test URL
          exposed;
      (e) shutdown() stops every adapter exactly once;
      (f) regression: end-to-end /_test/inject → /_test/last-reply
          round-trip records both authorised replies and unauthorised
          refusals (the secondary-bug guard).
- [x] Slim `src/index.ts` to ~30 LOC — just read env, call
      `runBridge`, wire SIGINT/SIGTERM to the shutdown handle.
- [x] Manual smoke: BRIDGE_E2E_TEST=1 daemon with two fake-token
      agents — bridge survives both TokenInvalid errors, CLI inject
      via `scripts/cerase-acp-cli inject --remote …` returns the
      fake-child reply for authorised users, the Italian refusal for
      unauthorised, and HTTP 500 for unknown agents (correct: an
      unknown agent is a programmer error from the ingress, not a
      user-facing condition).

**Exit gate**: `npm test` green (93/93); in test mode with all-fake
tokens the test server is reachable on :7474, inject round-trips
work, and shutdown is clean. In production mode the existing
fail-fast on token error is preserved with a best-effort teardown.
✅ CLOSED 2026-05-24.

**Effort**: 0.25 day → bumped to ~0.4 day because a secondary bug
surfaced during the manual smoke. Net: still cheaper than discovering
either bug in the cerase repo's e2e-discord tier.

## Out of scope for v0.1 (deferred)

- **Slack / Telegram adapters** — Tier-0 only, by customer request.
- **Multi-bot per agent** — one bot per template is the PoC contract.
- **Persistent session-store across container redeploys** — v0.1
  relies on mem0 + the persisted workspace inside the agent container
  to recover continuity on the next user turn. A dedicated
  session-store (à la OpenACP) is a v0.2 concern.
- **Skill command pinning, slash commands, button-based permission
  approval** — explicit UX-incompatibility per cerase M2 rules.
- **Doctor checks** — deferred; structured pino logs cover the PoC
  observability need.
